Read this doc top to bottom. Cross-references to the original proposal call out section names so the reader can pull design tokens, OKLCH risk values, or the wireframe-side `app.jsx` patterns when needed.

## Contract this plan reads from

The frontend reads from the FastAPI service inside `apx-demo/src/fraud_analyst/backend/`. The backend's response shapes are defined by the Pydantic models the parallel agent is implementing against `demo-client-graph-backend.md`.

The five endpoints the frontend consumes:

1. `POST /api/search/rings` returns a list of `RingOut`. Drives Screen 1 mode "Fraud rings".
2. `POST /api/search/risk` returns a list of `RiskAccountOut`. Drives Screen 1 mode "Risky accounts".
3. `POST /api/search/hubs` returns a list of `HubAccountOut`. Drives Screen 1 mode "Central accounts".
4. `POST /api/load` returns a `LoadOut`. Drives Screen 2 with a 7-step progress animation.
5. `POST /api/genie/ask` returns an `AskOut`. Drives Screen 3 conversational pane and the report modal.

The exact field names and types live in `demo-client-graph.md` lines 221 through 323 and are duplicated, with their pipeline source, in the backend plan's "Locked-in data contract" section. One field, `HubAccountOut.shortest_paths`, is mocked in the web service per the backend plan; the frontend treats it as a number like any other and does not need to know it is mocked.

The OpenAPI client at `src/fraud_analyst/ui/lib/api.ts` is auto-regenerated by apx whenever the backend route definitions change. The frontend does not regenerate it by hand. It does call the apx `refresh_openapi` MCP tool once after the backend agent reports the routes are stable, just to confirm the typed client is up to date before screen work starts.

## Frontend work items

Eleven items, sized to land as one commit each. Items F1 and F2 are setup that everything else depends on; do them first and in order. After that, F3 through F5 build the design-system primitives and helpers the screens consume; treat those as a parallel cluster. F6 through F8 are the three screens, in order. F9 through F11 are integration and polish.

### F1. Tailwind theme tokens and font wiring

**Why:** The wireframe's clean light aesthetic depends on a small set of named tokens, the warm off-white canvas, the OKLCH risk palette, the IBM Plex pairing, the 1px borders, the 3 to 4 px corner radii. Hardcoding hex values into components throws that away. Putting the tokens in the Tailwind config once and addressing them as utility classes keeps every later component cheap to write and visually consistent without inspection.

**What changes:** Extend the Tailwind config in the apx scaffold with the named color tokens, the IBM Plex font families, the smaller default border radius, and the 14 px base font size with 1.5 line height. The token values are listed in full in `demo-client-graph.md` lines 173 through 215. Use that block verbatim for color, font, border-radius, and font-size values.

Wire the IBM Plex fonts into `src/fraud_analyst/ui/index.html` as Google Fonts link tags, IBM Plex Sans weights 400, 500, 600, and IBM Plex Mono weights 400, 500.

In `src/fraud_analyst/ui/styles/globals.css`, set the body background to the canvas token, the body text color to the ink token, the default font family to IBM Plex Sans, and add a global `font-feature-settings` rule that turns on tabular numerals so tables align without per-element overrides.

**Verification:** Run `apx dev start`, open the dev URL, confirm the page background is the warm off-white, confirm the IBM Plex weight 400 renders in body text, confirm a placeholder element styled with `font-mono` renders in IBM Plex Mono.

### F2. Install shadcn primitives

**Why:** The wireframe uses interactive surfaces that have direct shadcn equivalents: buttons, cards, tables, checkboxes, selects, inputs, badges, dialogs, textareas, skeletons, tooltips. Generating these once into the project's components folder gives the screens a typed, accessible baseline; writing them by hand would burn time on details the design has no opinion on.

**What changes:** Install the eleven shadcn primitives listed in `demo-client-graph.md` line 89 via the apx `add_component` MCP tool. After install, confirm each component file ends up under `src/fraud_analyst/ui/components/ui/`. If any land elsewhere, move them.

The radio-card "What are you looking for?" choices on Screen 1 are visually distinctive enough to stay bespoke; do not add a shadcn radio group for them.

**Verification:** Run `apx dev check` and confirm zero TypeScript errors after the eleven components are present.

### F3. Risk color and ring layout helpers

**Why:** The risk palette is referenced by every results table, every thumbnail, every inline pill on every screen. The ring thumbnail layout uses a deterministic seed function so a given ring renders identically on every load and across analyst sessions. Centralizing both keeps the visual language consistent and the rendering cheap.

**What changes:** Create a `lib/riskColors.ts` exporting the three OKLCH risk colors keyed by H, M, L, plus a small helper that maps a risk score float to one of those three letters using the same thresholds the wireframe applies (high above 0.75, medium between 0.5 and 0.75, low below 0.5; confirm against any thresholds the backend's `RingOut.risk` field already encodes and align if the backend disagrees).

Create a `lib/ringLayout.ts` that ports the deterministic seed-based layout from the wireframe's `app.jsx`. The original is referenced in `demo-client-graph.md` lines 555 through 597 as the `ringSeed` and `ringLayout` functions. Port them to TypeScript with explicit types for ring inputs and node and edge outputs. Do not change the layout math; the goal is a faithful re-render.

**Verification:** Add a tiny unit test that calls `ringLayout` twice on the same ring id and asserts the output node positions are identical. This guards against accidental sources of nondeterminism creeping in later.

### F4. Bespoke SVG components, RingThumb and NetworkPreview

**Why:** These two components are the load-bearing visual moments on Screen 1. Pure SVG keeps them light; the alternative, a layout engine like Cytoscape, was explicitly rejected in the original proposal at `demo-client-graph.md` line 809 and remains the right call for the demo.

**What changes:** Implement `components/RingThumb.tsx` as a small SVG that takes a `RingOut` and renders the layout from F3 with risk-colored edges and nodes, hub nodes drawn larger and saturated. The dimensions, stroke widths, opacities, and selected-vs-unselected color handling are all specified in `demo-client-graph.md` lines 566 through 597; match them.

Implement `components/NetworkPreview.tsx` as a grid of small ring tiles, each tile reusing `RingThumb`. Clicking a tile selects that ring and the parent component is responsible for the selection state. Pass the selection in via props rather than reaching into a global store.

Both components stay presentational. They take typed data and render. They do not fetch.

**Verification:** Render each component in Storybook style by adding a temporary stub route that mounts them with a hand-crafted `RingOut`. Walk the four risk and topology combinations from the wireframe to make sure each renders without warnings. Remove the stub route before commit.

### F5. Building blocks, Pill, RiskBar, Stepper, Shell

**Why:** These four show up on every screen. Defining them once with the F1 tokens guarantees the pages compose cleanly.

**What changes:** Implement four small components.

`components/Pill.tsx` is the rounded inline tag used throughout the wireframe for status badges, identifier chips, and filter values. Takes a label and an optional intent (default, accent, risk-high, risk-med, risk-low) and renders one element using F1 tokens.

`components/RiskBar.tsx` is the horizontal bar showing a 0-to-1 score with a colored fill. Takes a score and a risk letter; renders a bordered track with a filled portion using the risk color.

`components/Stepper.tsx` is the top-of-page Search to Load to Analyze indicator. Takes the current step and a click handler; renders three numbered steps with the active step highlighted using the accent token. Click handler enables the wireframe's "jump between steps" behavior.

`components/Shell.tsx` is the page wrapper. Takes the current step, the click handler, the user identity string, and children. Renders the header with logo and user, the stepper, and the page body.

**Verification:** Mount each into the temporary stub route from F4 and walk the visual states. Run `apx dev check`. Remove the stub route.

### F6. Screen 1, search and find

**Why:** This screen is where the analyst spends the most time and is the most visually demanding. Doing it first means the design and the data shape both get exercised end to end before Screens 2 and 3 add behavior on top.

**What changes:** Create `routes/search.tsx` (or whatever route path matches the apx TanStack Router conventions already in `routes/__root.tsx`). The screen composes, top to bottom:

1. The "What are you looking for?" card with three custom radio-style choice cards: Fraud rings, Risky accounts, Central accounts. Selection drives which result set renders below.
2. The Filters strip with date range, minimum amount, and max nodes. Use shadcn `Select` for date range, shadcn `Input` for the two numeric fields. The values feed the API request body.
3. The results region. Branches on the selected mode:
   - Fraud rings mode renders an optional collapsible NetworkPreview above a results table whose rows each include a RingThumb. Each row also shows ring id, node count, volume, shared identifier pills, risk pill, and a checkbox to mark the ring for loading.
   - Risky accounts mode renders a single results table with account id, RiskBar, velocity pill, merchant diversity pill, and account age in days.
   - Central accounts mode renders a single results table with account id, neighbors count, betweenness score, shortest paths count.
4. A primary "Continue to Load" button at the bottom that is enabled only when at least one ring (in fraud rings mode) is checked. The button advances the stepper to step 2 and carries the selected ring ids forward.

API calls go through the auto-generated client. Use Suspense plus an ErrorBoundary at the route boundary so the screen handles loading and error states without per-component plumbing. The apx `get_route_info` MCP tool returns the exact hook name and the Suspense pattern for each generated endpoint; call it before writing the data fetching glue.

For empty states (no results returned), render a small SVG placeholder consistent with the wireframe's empty state.

**Verification:** Use the Playwright MCP server to script a walk: change mode three times, set filter values, run the search, select two rings, click Continue, confirm the stepper advances. Run `apx dev check`.

### F7. Screen 2, load to lakehouse

**Why:** The wireframe's load animation sets the rhythm of the demo. The backend returns the full `LoadOut` immediately; the visual choreography is the frontend's job. This decouples demo timing from server speed.

**What changes:** Create `routes/load.tsx`. On mount, the screen calls `POST /api/load` with the selected ring ids carried from Screen 1, receives the seven-step `LoadOut`, then walks through the steps client-side on a 700 ms timer. The wireframe's progress states are todo, now, done; rotate each step through them on the timer.

Below the steps, render two static-but-live blocks: the row counts from the `LoadOut.row_counts` field as a small definition list, and the quality checks from `LoadOut.quality_checks` as a list of pass/fail indicators with the friendly check name.

Two buttons at the bottom: a secondary "Back" that returns to Screen 1 keeping the selection intact, and a primary "Continue to Analyze" that becomes enabled once the seventh step lands in the done state.

**Verification:** Playwright walk: Continue from Screen 1, watch the seven steps animate, confirm the Continue button enables on completion, click through to Screen 3.

### F8. Screen 3, analyze with Genie

**Why:** The conversational pane is the demo's payoff. Most of the wiring is straightforward; the careful part is the typed Genie response with optional table.

**What changes:** Create `routes/analyze.tsx`. Two-pane layout:

Left pane is a sidebar with two sections. The first is a schema reference rendered as `<details>` elements, one per gold table, each listing the columns the analyst can ask about. The second is a list of sample-question buttons taken from the wireframe's canned set in `app.jsx`. Clicking a button puts that question into the input below and submits.

Right pane is a transcript. Each turn is a question bubble followed by an answer bubble. The answer bubble contains the `AskOut.text` and, if `AskOut.table` is present, a shadcn `Table` rendering the headers and rows. Below the transcript is a textarea input plus an "Ask" button. Pressing Enter or clicking the button calls `POST /api/genie/ask`, threading the `conversation_id` through subsequent calls so Genie keeps context.

Top-right of the screen has an "Export Report" button that opens the report modal from F9.

Skeleton states cover the in-flight question; the apx `get_route_info` Suspense pattern from Screen 1 applies here too.

**Verification:** Playwright walk: click two of the sample questions, confirm both produce a transcript bubble, confirm the second call carries the conversation id from the first, type a free-text question, confirm it appends correctly. Run `apx dev check`.

### F9. Report modal

**Why:** The wireframe ends with an export action. Wiring it makes the demo feel complete and gives the analyst a tangible artifact.

**What changes:** Implement `components/ReportModal.tsx` using the shadcn `Dialog`. The modal contains a structured report with the wireframe's report sections: a header with selected ring ids and a generated timestamp, a summary block, and a conversation log mirroring the Screen 3 transcript.

Two action buttons at the modal foot: "Print to PDF" calls the browser print API. "Save to lakehouse" calls a backend endpoint named `POST /api/reports` that the backend agent adds at the same time. If that endpoint is not yet available, leave the button disabled with a tooltip indicating the endpoint is pending; do not stub it client-side.

**Verification:** Open the modal from Screen 3, click Print to PDF, confirm the browser print dialog opens with the report layout intact. Click Save to lakehouse, confirm a success or pending state is communicated clearly.

### F10. Top-level routing and stepper navigation

**Why:** The three screens together form a guided flow. The user can also jump back via the stepper. URL-backed routing makes the flow shareable and refresh-safe.

**What changes:** Wire the three screens into the TanStack Router tree under the apx scaffold's `routes/__root.tsx`. Replace the placeholder `routes/index.tsx` with a redirect to the first step.

The Shell component from F5 receives the current step from the active route and a click handler that routes to the chosen step. Selected ring ids and the conversation id persist via the router's search params or a small zustand store, whichever the apx scaffold already prefers; check the scaffold's existing patterns before introducing a new state library.

**Verification:** Refresh on each of the three screens; confirm state survives the refresh where it should and resets cleanly where it should not. Click the stepper at every step to jump backward and forward. Run `apx dev check`.

### F11. Wire to the OpenAPI client and final smoke walk

**Why:** Up to this point, work items can pretend the data is real even when the backend services are still mocked. Item F11 confirms the generated typed client compiles against the real route definitions and that every screen survives a single end-to-end walk.

**What changes:** Run the apx `refresh_openapi` MCP tool. Open the regenerated `src/fraud_analyst/ui/lib/api.ts` and confirm the five endpoint hooks the screens use are present and typed.

Visit each screen in turn. Where any screen still calls a hand-rolled fetch or a mock, replace it with the generated hook. Remove any leftover mock service files or constants that were used as placeholders during F6 through F8.

Run `apx dev check` and confirm zero TypeScript errors. Do a final Playwright walk through all three screens end to end against the running backend, with the backend pointed at whichever services are real at this moment (Genie is real from day one per the backend plan; Neo4j, Delta, and the Genie ask each go from mock to real on independent timelines).

**Verification:** A clean check pass plus a clean Playwright walk across all three screens with no console errors.

## Hand-off boundary, what the frontend agent should not touch

- Anything inside `finance-genie/automated/`. That directory is the data pipeline. Pipeline work is owned by a separate agent against `demo-client-graph-backend.md`.
- Anything inside `apx-demo/src/fraud_analyst/backend/` other than reading the Pydantic field names to confirm the contract. Field names there are the contract; the frontend agent does not edit them.
- The `app.yml` resource bindings, the `databricks.yml` deploy targets, and the secret scope wiring. The original proposal at `demo-client-graph.md` line 99 lists those as operational items and they live with the backend agent.
- The Genie Space provisioning. The existing AFTER-GDS Genie Space is what the backend's `/api/genie/ask` endpoint targets. The frontend treats the endpoint as a black box.

If a work item above forces a backend or pipeline change, stop and raise it. The contract in `demo-client-graph-backend.md` and the route shapes in the backend `models.py` are meant to be sufficient; if they are not, those documents change first, then the frontend follows.

## Order of operations

1. F1 first. Without the design tokens, every later component has to be revisited.
2. F2 next, in parallel if helpful with F1's font wiring step.
3. F3, F4, F5 as a cluster. They have no interdependencies; pick whatever order fits.
4. F6, F7, F8 in order. Each screen builds on the previous via the stepper and the carried selection state.
5. F9 attaches to Screen 3 and can land at the same commit or one commit later, the screen is complete without it for early demos.
6. F10 lands once at least Screens 1 and 2 exist, so the routing has somewhere to go.
7. F11 lands last, after the backend agent reports routes are stable.

The frontend agent does not block on the pipeline agent at any point. The backend agent's mock services keep every screen renderable from F6 onward; the swap to real data happens behind the OpenAPI client and is invisible to the React code.
