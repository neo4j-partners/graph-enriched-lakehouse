# Graph Enrichment and Databricks Genie

**Summary:** Databricks Genie is becoming the default analyst interface to the Lakehouse, but it cannot answer questions about how entities connect because those relationships do not live in flat rows. Neo4j runs graph algorithms against the Lakehouse and writes structural scores back as Delta columns, so Genie answers relationship questions as ordinary SQL with no change to the tool or the analyst workflow.

Databricks is centering its go-to-market this year on Genie, a natural language BI service that translates analyst questions into SQL against Lakehouse tables. Genie is on track to become the default way enterprises consume governed Lakehouse data, which makes every Genie deployment an opportunity for Neo4j to extend the question surface analysts can reach. Genie handles standard BI operations well, but it is limited to what flat rows can express. Questions about how entities connect do not live in rows: communities, influence, multi-hop paths, and shared neighborhoods cannot be answered well in SQL no matter how good the language model becomes. Neo4j closes that gap. Graph algorithms run against the relationships already implicit in the Lakehouse and write structural scores back as plain Delta columns. Genie queries those columns like any other dimension, with no changes to the tool and no retraining for the analyst. Each platform plays to its strength: Neo4j for graph computation, Databricks for governed storage and the analyst interface. The result is a lightweight, repeatable integration that attaches Neo4j to the flagship Databricks interface rather than competing with it.
